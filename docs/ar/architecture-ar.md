# البنية المعمارية

## نظرة عامة على النظام

يتبع نظام CP'S Enterprise DCS بنية وكلاء ذكية ثلاثية الطبقات حيث تعمل وكلاء ذاتية على المستويات المحلية والإقليمية والعالمية. لكل طبقة مسؤوليات متميزة وتتواصل عبر gRPC مع Protocol Buffers.

```
                    ┌───────────────────────┐
                    │    Master Agent        │
                    │  (Global Orchestration)│
                    └───────────┬───────────┘
                                │
                ┌───────────────┼───────────────┐
                ▼                               ▼
    ┌───────────────────┐           ┌───────────────────┐
    │  Regional Agent   │           │  Regional Agent   │
    │  (Go / Raft)      │           │  (Go / Raft)      │
    └───────┬───────────┘           └───────┬───────────┘
```

## تفاصيل المكونات

### 1. واجهة نقطة البيع (React/TypeScript)

**الموقع:** `cps-enterprise-dcs/pos-interface/`

الواجهة الأمامية لنقطة البيع التي يتفاعل معها الكاشير مباشرة. مبنية بـ React 18 و TypeScript و Zustand لإدارة الحالة.

**الميزات الرئيسية:**
- شبكة منتجات مع تصفية حسب الفئة وبحث
- سلة تسوق مع حساب ضريبي فوري
- إدارة الجلسات (البدء/الإنهاء مع تتبع الرصيد النقدي)
- قدرة العمل دون اتصال مع مؤشر حالة المزامنة
- التواصل مع الوكيل المحلي عبر gRPC-Web

**إدارة الحالة:** تتعامل مخازن Zustand مع حالة الجلسة وحالة السلة وحالة المزامنة. صُمّم المخزن لإدراج المعاملات في طابور محلي عند انقطاع الاتصال ومزامنتها عند استعادة الاتصال.

### 2. تطبيق الإدارة (React/TypeScript)

**الموقع:** `app/`

لوحة الإدارة المبنية بـ React 19 و Vite 7 ومكونات shadcn/ui. توفر واجهات إدارة لنظام التجارة.

**مكتبة الواجهة:** تستخدم مجموعة شاملة من عناصر Radix UI الأولية عبر shadcn/ui، بما في ذلك مربعات الحوار والقوائم المنسدلة والتبويبات والأكورديونات وعرض البيانات عبر Recharts.

### 3. الوكيل المحلي (Python)

**الموقع:** `cps-enterprise-dcs/local-agent/`

وكيل الحوسبة الطرفية الذي يعمل في كل موقع فرع/متجر. يعمل بشكل ذاتي حتى أثناء العزلة الكاملة عن الشبكة.

**المسؤوليات:**
- **مخزن الأحداث**: يحتفظ بمخزن أحداث محلي SQLite لجميع الأحداث المالية
- **إدارة CRDT**: ينفذ GCounter و PNCounter و ORSet و LWWRegister لتكرار البيانات بدون تعارض
- **خادم gRPC**: يعرض خدمات لواجهة نقطة البيع لتقديم المعاملات
- **الأمان**: تشفير المغلفات (AES-256-GCM) لجميع الأحداث المالية
- **المزامنة**: يزامن الأحداث وحالات CRDT دورياً مع الوكيل الإقليمي

**الوحدات الرئيسية:**
| الوحدة | الغرض |
|--------|---------|
| `agent.py` | دورة حياة الوكيل الأساسية والتنسيق |
| `crdt.py` | تنفيذات CRDT (GCounter, PNCounter, ORSet, LWWRegister) |
| `event_store.py` | مخزن أحداث للإلحاق فقط مبني على SQLite |
| `grpc_server.py` | تنفيذ خدمة gRPC |
| `security.py` | تشفير المغلفات وتوقيع HMAC |

### 4. الوكيل الإقليمي (Go)

**الموقع:** `cps-enterprise-dcs/regional-agent/`

وكيل التنسيق الإقليمي الذي يجمع البيانات من عدة وكلاء محليين باستخدام إجماع Raft.

**المسؤوليات:**
- **إجماع Raft**: يستخدم HashiCorp Raft لانتخاب القائد ونسخ السجلات عبر العقد الإقليمية
- **تجميع CRDT**: يدمج حالات CRDT من جميع الوكلاء المحليين في المنطقة
- **نشر الأحداث**: يمرر الأحداث إلى مخزن أحداث PostgreSQL
- **خادم gRPC**: يقبل الاتصالات من الوكلاء المحليين ويوفر خدمات التنسيق

**الحزم الداخلية:**
| الحزمة | الغرض |
|---------|---------|
| `internal/agent/` | وكيل مبني على Raft مع منطق التشغيل والانضمام والإجماع |
| `internal/config/` | تحليل التكوين من البيئة والأعلام |
| `internal/crdt/` | مدير CRDT مع عمليات الدمج |
| `internal/server/` | خادم gRPC للتواصل بين الوكلاء |

## مصدر الأحداث (Event Sourcing)

تُسجَّل جميع تغييرات الحالة في DCS كأحداث غير قابلة للتعديل. مخزن الأحداث هو مصدر الحقيقة الوحيد.

### تدفق الأحداث

```
POS → Local Agent → SQLite (local) → Regional Agent → PostgreSQL (regional)
                                                     → Kafka (streaming)
```

### بنية الحدث

كل `SovereignFinancialEvent` يحتوي على:
- **المعرّفات**: `event_id` (UUID v7) و `correlation_id` و `idempotency_key`
- **السياق**: `branch_id` و `agent_id` و `tenant_id`
- **المحتوى**: `type` (تعداد EventType) و `amount` و `currency` و `payload` المشفّرة
- **السببية**: `CausalContext` مع ساعة منطقية هجينة وساعة متجهية
- **التدقيق**: `created_by` و `approved_by` و `digital_signature` (ECDSA)

### أنواع الأحداث

تُصنف الأحداث في تدفقات:
- **المبيعات**: `SALE_INITIATED` و `SALE_COMPLETED` و `SALE_REVERSED` و `SALE_PARTIALLY_REFUNDED`
- **المخزون**: `INVENTORY_RECEIVED` و `INVENTORY_SOLD` و `INVENTORY_TRANSFERRED_*` و `LOW_STOCK_ALERT`
- **المدفوعات**: `PAYMENT_AUTHORIZED` و `PAYMENT_CAPTURED` و `PAYMENT_REFUNDED` و `PAYMENT_FAILED`
- **المحاسبة**: `LEDGER_ENTRY_POSTED` و `END_OF_DAY_RECONCILIATION` و `PERIOD_CLOSED`
- **Saga**: `SAGA_INITIATED` و `SAGA_STEP_COMPLETED` و `SAGA_COMPENSATION_TRIGGERED`
- **الوكيل**: `AGENT_DECISION_MADE` و `AGENT_ACTION_EXECUTED` و `AGENT_LEARNING_UPDATED`

## أنواع البيانات المتقاربة بدون تعارض (CRDTs)

تضمن CRDTs تقارب الحالة الموزعة دون الحاجة إلى تنسيق. يُنفذ DCS أربعة أنواع من CRDT:

| CRDT | حالة الاستخدام | استراتيجية الدمج |
|------|----------|----------------|
| **GCounter** | عدّ المعاملات وعدّ الزيارات | الحد الأقصى لكل عقدة |
| **PNCounter** | كميات المخزون (الإضافة/الإنقاص) | الحد الأقصى لكل عقدة لكل من P و N |
| **ORSet** | كتالوجات المنتجات النشطة وجلسات المستخدمين | اتحاد مع تتبع علامات الإزالة |
| **LWWRegister** | أسعار المنتجات وقيم الإعدادات | يفوز أحدث طابع زمني |

### المزامنة

1. يحتفظ الوكيل المحلي بحالة CRDT محلياً
2. عند حلول دورة المزامنة (افتراضياً 30 ثانية)، يرسل الوكيل المحلي حزم حالة CRDT إلى الوكيل الإقليمي
3. يدمج الوكيل الإقليمي الحالات من جميع الوكلاء المحليين باستخدام دوال دمج خاصة بكل نوع
4. تُنشر الحالة المدموجة إلى جميع الوكلاء المحليين

## الإجماع (Raft)

يستخدم الوكيل الإقليمي بروتوكول إجماع Raft من أجل:

- **انتخاب القائد**: تصبح إحدى العقد الإقليمية القائد، والباقي تابعون
- **نسخ السجل**: يقوم القائد بنسخ سجلات الأحداث إلى جميع التابعين
- **التحمل ضد الأعطال**: يستمر العنقود في العمل طالما أن أغلبية العقد على قيد الحياة

**التنفيذ**: يستخدم مكتبة HashiCorp Raft مع BoltDB للتخزين المستقر.

## نموذج الأمان

### تشفير المغلفات

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Plaintext   │────▶│  AES-256-GCM │────▶│  Ciphertext   │
│  Event Data  │     │  (DEK)       │     │  + Auth Tag   │
└──────────────┘     └──────┬───────┘     └──────────────┘
                            │
                     ┌──────▼───────┐
                     │  Vault KEK   │
                     │  (Transit)   │
                     └──────────────┘
```

1. يُنشأ **مفتاح تشفير بيانات (DEK)** فريد لكل حدث
2. يقوم DEK بتشفير حمولة الحدث باستخدام **AES-256-GCM**
3. يُشفَّر DEK نفسه بواسطة **مفتاح تشفير مفاتيح (KEK)** مخزَّن في HashiCorp Vault
4. تُحسب توقيع **HMAC-SHA512** على البيانات الوصفية الظاهرة للتحقق من السلامة
5. يُرفق **توقيع رقمي ECDSA** من قبل منشئ الحدث لعدم الإنكار

### الامتثال بدون كشف معرفة

يمكن توليد إثباتات الامتثال دون كشف البيانات الأساسية، باستخدام حقول إثبات بدون كشف معرفة في رسالة `SovereignPayload`.

## مخطط قاعدة البيانات

يشمل مخزن أحداث PostgreSQL (`cps-enterprise-dcs/event-store/schema.sql`):

| الجدول | الغرض |
|-------|---------|
| `event_store` | سجل أحداث للإلحاق فقط مقسم (8 أقسام تجزئة) |
| `stream_metadata` | تتبع نسخة التدفق للتحكم بالتزامن المتفائل |
| `snapshots` | لقطات تجميعية لتحسين القراءة |
| `saga_instances` | آلات حالة المعاملات الموزعة |
| `saga_steps` | تتبع خطوات Saga الفردية |
| `projection_sales_summary` | عرض مادي يومي للمبيعات |
| `projection_inventory` | مستويات المخزون الفورية |
| `projection_customer_loyalty` | نقاط ولاء العملاء والمستويات |
| `crdt_state` | حالات CRDT المحفوظة لكل عقدة |
| `agent_registry` | اكتشاف الوكلاء وتتبع الصحة |
| `audit_log` | سجل تدقيق مقسم شهرياً |

يفرض أمان مستوى الصف (RLS) عزل المستأجر والفرع.
